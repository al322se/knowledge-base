# База знаний

Markdown-only база знаний для связанных репозиториев, технических решений и бизнесового контекста.

Репозиторий предполагается хранить рядом с репозиториями, которые он описывает:

```text
рабочая-папка/
  knowledge-base/
  api-gateway/
  auth-service/
  frontend-app/
  worker-service/
```

В проекте нет кода, сервисов, пакетов, баз данных, скриптов и генераторов. Агент читает [инструкции](AGENTS.md), проходит по соседним репозиториям и источникам знаний, затем поддерживает структурированные markdown-файлы.

## Главная идея

Не делать один огромный файл знаний.

Использовать постепенное раскрытие:

1. [Индекс базы знаний](KB_INDEX.md) - короткая карта.
2. [Инвентаризация репозиториев](инвентаризация/репозитории.md) - список источников кода и их версий.
3. [Карты репозиториев](репозитории/ОПИСАНИЕ.md) - короткие карточки реп.
4. [Области знаний](области-знаний/ОПИСАНИЕ.md) - бизнесовые и технические домены.
5. [Бизнесовые знания](бизнес/ОПИСАНИЕ.md) - правила, процессы, роли, термины, ограничения.
6. [Архитектура](архитектура/обзор.md) - зависимости, потоки, решения.
7. [Источники](источники/ОПИСАНИЕ.md) - Confluence, треды митзоны, транскрипты, документы, ответы пользователя.

## Первый запуск

Попроси агента выполнить [первую проходку](процесс/первая-проходка.md).

После первой проходки должны появиться или обновиться:

- `инвентаризация/репозитории.md`
- `репозитории/<имя-репозитория>.md`
- `области-знаний/<область>.md`
- `архитектура/зависимости-репозиториев.md`
- `решения/требует-уточнения.md`
- `глоссарий.md`
- `ссылки.md`

## Публичность

Репозиторий может быть публичным, но конкретные карты репозиториев, бизнесовые правила, локальные пути, ссылки на Confluence и треды могут раскрывать приватный контекст. Публикуй такие страницы только намеренно.
